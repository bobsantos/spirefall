# Add Pause Button to HUD Top Bar

## Context and Problem Statement

Spirefall currently relies on the Escape key to open the pause menu (handled by `Game.gd._unhandled_input`). On mobile devices there is no Escape key, so players have no way to pause, access settings, or quit to the main menu during gameplay. A visible, touch-friendly pause control must be added to the HUD.

## Decision Drivers

* Mobile players have no physical Escape key and need a touch target to pause
* The pause button must meet minimum touch target size (64x64) on mobile for reliable input
* The HUD top bar already contains action buttons (SpeedButton, CodexButton) and is the natural location for game controls
* The implementation should reuse the existing `GameManager.toggle_pause()` and `PauseMenu` infrastructure with no new systems

## Considered Options

* Top bar button (rightmost position after CodexButton)
* Floating button (fixed position, overlays the playfield)
* Swipe/tap gesture (e.g., three-finger tap or swipe-down from top)

## Decision Outcome

Chosen option: "Top bar button", because it is consistent with the existing SpeedButton and CodexButton pattern, discoverable without a tutorial, and does not obscure gameplay or conflict with tower placement gestures.

### Consequences

* Good, because it reuses the existing TopBar HBoxContainer layout and `GameManager.toggle_pause()` signal flow
* Good, because the PauseMenu already listens to `paused_changed` so no additional wiring is needed
* Good, because the button is always visible and trivially discoverable
* Bad, because it adds one more element to the top bar, slightly reducing horizontal space for info labels on small screens. Mitigated by allowing labels to compress/truncate while action buttons maintain minimum sizes.

### Confirmation

* Unit test `tests/unit/ui/test_hud_pause_button.gd` validates button existence, signal wiring, positioning, and mobile sizing
* Manual playtesting on mobile viewport confirms touch target is large enough

## Pros and Cons of the Options

### Top bar button

* Good, because consistent with existing button pattern (SpeedButton, CodexButton)
* Good, because always visible; no hidden gesture to discover
* Good, because minimal implementation (add node + connect signal)
* Bad, because takes horizontal space in an already populated top bar

### Floating button

* Good, because independent of top bar layout; can be positioned anywhere
* Bad, because overlays the playfield, potentially obscuring towers or enemies
* Bad, because inconsistent with the other action buttons in the top bar
* Bad, because requires z-order management to avoid being hidden behind panels

### Swipe/tap gesture

* Good, because takes zero screen space
* Bad, because undiscoverable without a tutorial or onboarding prompt
* Bad, because risk of accidental triggers during tower placement or drag interactions
* Bad, because requires gesture recognition system that does not exist yet

## More Information

The pause button only needs to trigger pause, not unpause. The PauseMenu Resume button handles unpausing. Therefore the button does not need `PROCESS_MODE_WHEN_PAUSED`. On desktop the button text is "||" (pause symbol), matching the simple text style of SpeedButton ("1x") and CodexButton ("Codex (C)"). Desktop size is 40x40; mobile size is `UIManager.MOBILE_BUTTON_MIN` (64x64).
